DDD의 Ubiquitous Language다. 정의보다 **강제**가 핵심이다 — 기획서, 대화, 코드,
DB 컬럼이 같은 단어를 쓴다. 단어가 바뀌면 코드도 따라 바뀐다.

각 항목마다 **코드 어디에 사는지**를 같이 적는다. 자기 구현을 가리키지 못하는
용어는 이미 표류했다.

이 프로젝트는 이중언어다. 도메인은 한국 개인 재무고 코드는 영어다. `청구주기`와
`billing cycle`을 못박아 두지 않으면 둘은 천천히 다른 개념으로 갈라진다.

<Glossary>
  <Term term="units" ko="단위 / 사실" code="posting.units_minor">
    무엇이 움직였나. 자기 통화 minor unit의 정수. USD 계정 `-75000`은 −$750.00.
  </Term>

  <Term term="weight" ko="가중치 / 측정" code="posting.weight_minor" notTo="units와 절대 혼동 말 것. 같은 사건의 서로 다른 두 표현이다.">
    같은 사건을 기능통화(KRW)로 옮긴 값. 균형 검사는 이것만 본다. 계정 KRW 장부가액은
    `SUM(weight_minor)`로 바로 나온다.
  </Term>

  <Term term="booking commodity" ko="기표통화" code="txn.booking_commodity" notTo="posting의 commodity와 다르다 — 그건 다리마다 다를 수 있다.">
    book의 기능통화. 항상 KRW. book마다 하나고, 바꾸려면 새 book을 연다.
  </Term>

  <Term term="plug" code="weight_source = 'plug'" notTo="반올림 오차가 아니다. 실제 비용이다.">
    어느 다리도 KRW가 아닐 때 남는 잔차. 에러가 아니라 **FX 스프레드**다. 손익계산서에
    올린다.
  </Term>

  <Term term="청구주기" ko="billing cycle" code="card.cycle_close_day">
    사용분이 어느 청구서에 실리고, 현금이 언제 빠지는지를 정하는 규칙. 없으면
    원장은 빚진 금액은 알아도 **언제** 나가는지는 모른다.
  </Term>

  <Term term="cash" ko="현금 계정" code="account.cash" notTo="asset과 다르다. 아파트도 자산이지만 현금이 아니다.">
    `holiday cashflow`가 잔고를 걸어가는 출발점. **접두어 규칙이 아니라 플래그**인 까닭은
    아래 ADR-004에 있다.
  </Term>

  <Term term="schedule" ko="스케줄" code="card / installment / recurring / loan" notTo="원장이 아니다. 절대 전기되지 않는다.">
    돈이 언제 나갈지에 대한 **예측**. 카드 청구주기·할부·정기지출·대출 네 종류가 있고,
    넷 모두 `cashRunway` 하나로 들어간다. 근사를 허용하는 유일한 구역이다.
  </Term>

  <Term term="checkpoint" ko="체크포인트" code="snapshot.kind = 'checkpoint'" notTo="close(마감)와 다르다. 저널을 잠그지도, 재평가를 전기하지도 않는다.">
    스냅샷만 찍는 소프트 마감. 일별·주별이 여기 속한다. hard close는 book당 그레인
    하나뿐이다 — 하루는 달 안에 들어가므로, 양쪽에서 재평가하면 FX가 이중 계상된다.
  </Term>
</Glossary>
