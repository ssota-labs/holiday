이 프로젝트는 **정책이 곧 제품**이다. 아래는 구현 세부가 아니라 숫자를 믿을 수 있는
이유 전부고, 각각이 **그걸 강제하는 테스트에 링크**돼 있다. 링크가 안 맞으면
`scripts/check-rule-links.ts`가 빌드를 죽인다 — 링크가 해결되지 않는 정책 문서는 없는
것보다 나쁘다. **증거처럼 읽히니까.**

이 문서를 "스펙"이라고 부르지 않는 이유: 스펙(PRD)은 *뭘 만들지*를 말한다. 이건 DDD
용어로 **Business Rules Catalog**와 **Ubiquitous Language**다.

스케줄(카드·할부·정기지출·대출) 관련 규칙은 [스케줄 네 개](/docs/v0.1/schedules)에 있다.

## 원장 규칙

<Rule
  id="POLICY-001"
  title="허용오차는 존재하지 않는다"
  test="packages/core/src/domain/txn.test.ts"
  testName="rejects an unbalanced transaction with the exact residual, and no tolerance"
  consequence="₩50 누락 수수료, off-by-one, 반올림 누수가 조용히 흡수된다 — 정확히 잡고 싶은 크기가."
>
전기들의 `weight_minor` 합은 **정확히 0**이다. 정수 산술이고, 시스템 어디에도 fuzz
파라미터가 없다.

환율을 저장해 상대금액을 유도하면 정수 스케일에서 lossy하고, 그 관례적 해법이
허용오차 발명이다. 그래서 상대금액을 사실로 저장하고 환율은 표시용으로 격하했다.
</Rule>

<Rule
  id="POLICY-002"
  title="KRW 전기의 weight는 units와 같아야 한다"
  test="packages/core/src/domain/txn.test.ts"
  testName="refuses a booking-commodity posting whose weight disagrees with its units"
  consequence="기능통화 다리가 자기 자신과 다른 값으로 측정될 수 있게 되고, 잔액이 조용히 표류한다."
>
`commodity === bookingCommodity`이면 `weight_minor === units_minor`. 도메인과 SQLite
트리거 양쪽에서 강제된다.
</Rule>

<Rule
  id="POLICY-004"
  title="결제와 환불은 상대 leg으로 구분한다"
  test="packages/core/src/domain/cashflow.test.ts"
  testName="does NOT let a payment of a previous bill cancel out this cycle"
  consequence="지난달 결제가 이번달 청구서를 상쇄해서 청구액이 과소계상된다 — 정확히 사람을 마이너스 통장으로 보내는 방향으로."
>
카드 계정에서 구매는 음수지만 **환불과 결제가 둘 다 양수**다. 순진하게 합산하면 안
된다. 결제의 상대 leg은 funding 계정이고 환불의 상대 leg은 비용 계정이다.
</Rule>

## 용어

<Glossary>
  <Term term="units" ko="단위 / 사실" code="posting.units_minor">
    무엇이 움직였나. 자기 통화의 minor unit 정수. USD 계정의 `-75000`은 −$750.00.
  </Term>

  <Term term="weight" ko="가중치 / 측정" code="posting.weight_minor" notTo="units와 절대 혼동 말 것. 같은 사건의 서로 다른 두 표현이다.">
    같은 사건을 기능통화(KRW)로 표현한 값. 균형은 이것만 본다. 계정의 KRW 장부가액은
    그냥 `SUM(weight_minor)`이고 공짜로 나온다.
  </Term>

  <Term term="booking commodity" ko="기표통화" code="txn.booking_commodity" notTo="posting의 commodity와 다르다 — 그건 다리마다 다를 수 있다.">
    book의 기능통화. 항상 KRW. book당 하나로 고정이고 바꾸려면 새 book이다.
  </Term>

  <Term term="plug" code="weight_source = 'plug'" notTo="반올림 오차가 아니다. 실제 비용이다.">
    어느 다리도 KRW가 아닐 때 남는 잔차. 에러가 아니라 **FX 스프레드**이고 손익계산서에
    올린다.
  </Term>

  <Term term="청구주기" ko="billing cycle" code="card.cycle_close_day">
    사용분이 어느 청구서에 실리고 언제 현금이 나가는지를 정하는 규칙. 이게 없으면
    원장은 얼마를 빚졌는지는 알아도 **언제** 나가는지를 모른다.
  </Term>

  <Term term="cash" ko="현금 계정" code="account.cash" notTo="asset과 다르다. 아파트도 자산이지만 현금이 아니다.">
    `holiday cashflow`가 잔고를 걸어가는 출발점. **접두어 규칙이 아니라 플래그**인 이유는
    아래 ADR-004에 있다.
  </Term>

  <Term term="schedule" ko="스케줄" code="card / installment / recurring / loan" notTo="원장이 아니다. 절대 전기되지 않는다.">
    언제 돈이 나갈지에 대한 **예측**. 카드 청구주기·할부·정기지출·대출 네 종류이고, 넷 다
    `cashRunway` 하나로 들어간다. 예측이라 근사가 허용되는 유일한 곳이다.
  </Term>

  <Term term="checkpoint" ko="체크포인트" code="snapshot.kind = 'checkpoint'" notTo="close(마감)와 다르다. 저널을 잠그지도, 재평가를 전기하지도 않는다.">
    스냅샷만 뜨는 소프트 마감. 일별·주별이 여기 해당한다. hard close는 book당 하나의
    그레인만 — 하루는 달 안에 있어서 양쪽에서 재평가하면 FX가 이중 계상된다.
  </Term>
</Glossary>

## 결정

<Decision
  id="ADR-001"
  title="환율이 아니라 상대금액으로 균형을 잡는다"
  status="locked"
  date="2026-07-17"
  commit="836f6c5"
  rejected={[
    { option: '거래마다 기표통화 + 전기마다 환율', because: '환율을 되곱하면 정수 스케일에서 lossy하다. ₩1,000,000 → $750.00은 1333.3333…이고 역산하면 ₩999,998. 그럼 허용오차를 발명하게 되고, 그게 잡고 싶은 크기를 가린다.' },
    { option: 'Beancount의 통화별 weight 그룹', because: '기준통화 없이 균형을 잡을 수 있는 대신 허용오차 추론과 booking 알고리즘이 필요하고, 무엇보다 KRW 장부가액이 명시되지 않아 매번 lossy한 변환 패스가 필요하다.' },
  ]}
>
전기마다 `units_minor`(사실)와 `weight_minor`(측정)를 저장한다. weight는 Beancount의
`@@`(총액 가격)를 환율에서 곱해내지 않고 직접 저장한 것이다.

**locked인 이유:** 5년치 전기가 "이 사건의 KRW 측정"을 인코딩한다. 기준통화를 옮기려면
과거 전부를 `units × rate`로 다시 유도해야 하는데, 그게 바로 이 설계가 피하려는 연산이다.
새 인생 국면은 마이그레이션이 아니라 새 book이다.
</Decision>

<Decision
  id="ADR-002"
  title="JSONL 저널을 버리고 SQLite를 원장으로, 감사는 해시 체인으로"
  status="accepted"
  date="2026-07-17"
  commit="74677cb"
  rejected={[
    { option: 'JSONL 저널을 SoR로 + git log -p를 감사 수단으로', because: '사용자가 git 커밋으로 감사하길 원치 않았다. 그러면 저널은 쿼리도 안 되고(rebuild는 순차 replay뿐) byte-identical 라운드트립을 스키마 버전 넘어 영원히 보장하는 대가만 남는다.' },
    { option: 'SQLite .db를 gitignore', because: '오프머신 백업과 "어제 파일로 되돌리기"를 잃는다. 바이너리라 diff는 안 나오지만, diff를 보는 건 애초에 git 감사였고 그건 뺐다.' },
  ]}
>
`.db`가 system of record이고 커밋된다. 감사는 `audit_log` 해시 체인 — 각 행이 직전 행에
커밋하고, 거래 행은 **내용 해시**에 커밋한다. 손으로 전기를 고치면 잔액은 맞아도
`verify`가 잡는다.

**정직한 한계:** 우발적 변조는 잡지만 체인 전체를 재계산하는 사람은 못 막는다. head를
파일 밖에 앵커해야 닫힌다 — `holiday verify --head`가 그 용도다.

저널과 달리 해시는 버전 dispatch가 된다. `CHAIN_HASH_VERSION`을 올리면 과거 행은 자기
버전으로 계속 검증된다. byte-stable 저널엔 없던 탈출구다.
</Decision>

<Decision
  id="ADR-004"
  title="현금은 계정 플래그다. 코드 접두어가 아니다"
  status="accepted"
  date="2026-07-17"
  commit="5036abe"
  rejected={[
    { option: '`Assets:Bank*` / `Assets:Cash*` 접두어 규칙', because: '관례를 사실인 척한 것이었다. 다른 이름 밑에 있는 현금이 투영에서 **조용히** 빠졌고, 그럼 숫자는 여전히 자신만만하게 틀린다. 틀리는 방식 중 최악이다.' },
  ]}
>
`account.cash`. 마이그레이션이 옛 관례에서 백필하고, `account add`와 `cashflow` 양쪽이
현금으로 표시되지 않은 자산 계정을 **소리내어 말한다**.

이걸 만들다가 더 심각한 버그를 찾았다: 마이그레이션을 `init` 명령만 실행하고 있어서,
기존 원장을 새 바이너리로 열면 없는 컬럼을 읽고 죽었다. ledger.db는 사용자가 커밋해두는
원장이고 플러그인 바이너리는 따로 업데이트되니, **플러그인을 한 번 업데이트하면 모든
기존 장부가 안 열렸을 것이다.** 새 DB로만 테스트했으면 못 봤다.
</Decision>

<Decision
  id="ADR-005"
  title="스키마는 drizzle-kit. 쿼리는 우리 것"
  status="accepted"
  date="2026-07-17"
  commit="75a75e8"
  rejected={[
    { option: 'drizzle-orm 전면 도입 (쿼리 포함)', because: 'Drizzle의 SQLite 세션은 setReadBigInts(true)를 절대 호출하지 않고 integer()가 내부적으로 `number int53`으로 타이핑돼 있다. 2^53을 넘으면 ERR_OUT_OF_RANGE로 죽고 주입할 훅이 없다(prepareQuery 안의 지역 변수). 이 원장의 금액은 i64이고 적합성 스위트가 2^53+1 왕복을 못박는다.' },
    { option: "blob({mode:'bigint'})", because: 'bigint를 십진수 ASCII로 BLOB에 저장한다. 정확하지만 SUM()이 안 되고 ORDER BY가 바이트 정렬이라 getBalances가 통째로 날아간다.' },
    { option: '손으로 쓴 SQL 유지', because: '마이그레이션 관리가 원시적이었고, 무엇보다 "적용된 마이그레이션 수정 금지"가 강제되지 않는 주석이었다.' },
  ]}
>
DDL은 Drizzle이 만들고 읽기는 `Db` 래퍼가 한다. `integer()`가 뱉는 DDL은 평범한
`INTEGER`라 이 분리는 비용이 0이다.

**도입 동기였던 전제는 성립하지 않았다.** Drizzle은 SQLite와 Postgres에 하나의 스키마를
주지 않는다 — `sqliteTable`과 `pgTable`은 별개 빌더고 컬럼 타입도 안 겹친다. Supabase는
어차피 자기 스키마를 쓴다. SQLite의 `RAISE(ABORT)`와 Postgres의 plpgsql `RAISE
EXCEPTION`은 방언이 다른 거지 툴체인 문제가 아니고, 포트 경계가 이미 그걸 허용한다.

**진짜 얻은 건 해시 추적이다.** `__holiday_migrations`가 각 마이그레이션의 sha256을
기록해서, 적용된 걸 수정하면 다음 열기에서 터진다. Drizzle의 `__drizzle_migrations`에서
가져온 아이디어인데 테이블은 우리 것이다 — Drizzle의 `migrate()`는 런타임에 디스크에서
`.sql`을 읽고, 이 CLI는 옆에 아무것도 없는 단일파일이다.

**대가:** STRICT 테이블을 잃었다(drizzle-kit 미지원). 손편집으로 INTEGER 컬럼에 TEXT를
넣는 걸 막던 ring 3 하나가 빠졌다. 그리고 RC(`1.0.0-rc.4`)에 핀됐다 — stable에는
node:sqlite 드라이버가 없다.
</Decision>
