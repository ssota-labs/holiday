---
title: ADR
description: 무엇을 골랐고, 무엇을 왜 버렸나
type: decision
summary: 거부한 대안이 본문이다. 코드를 읽어선 알 수 없는 부분이라서.
---

**각 항목의 본문은 거부한 대안이다.** 선택한 쪽은 코드만 읽어도 보인다.
시도했다가 버린 쪽은 코드에 남지 않는다. 그게 정확히 1년 뒤에 다시 올라온다.
"환율을 그냥 저장하면 안 되나?"에는 답이 있다. 그 답은 누군가 원장을 다시 쓰기 전에
찾을 수 있는 자리에 있어야 한다.

`status`는 되돌릴 수 있는지를 숨기지 않는다. `accepted`는 바뀔 수 있다. `locked`는
과거를 다시 쓰는 마이그레이션 없이는 못 바꾼다.

    { option: 'Beancount의 통화별 weight 그룹', because: '기준통화 없이 균형을 잡을 수 있는 대신 허용오차 추론과 booking 알고리즘이 필요하고, 무엇보다 KRW 장부가액이 명시되지 않아 매번 lossy한 변환 패스가 필요하다.' },
  ]}
>
전기마다 `units_minor`(사실)와 `weight_minor`(측정)를 둔다. weight는 Beancount
`@@`(총액 가격)를 환율에서 곱해 내지 않고 직접 적은 값이다.

**locked인 이유:** 5년치 전기가 "이 사건의 KRW 측정"을 이미 인코딩한다. 기준통화를
옮기려면 과거 전부를 `units × rate`로 다시 끌어내야 한다. 바로 이 설계가 피하려는
연산이다. 새 인생 국면은 마이그레이션이 아니라 새 book으로 연다.
</Decision>

>
`.db`가 system of record다. 커밋 대상이다. 감사는 `audit_log` 해시 체인이다. 각 행이

**정직한 한계:** 우발적 변조는 잡는다. 체인 전체를 다시 계산하는 사람은 못 막는다.
head를 파일 밖에 앵커해야 닫힌다. `holiday verify --head`가 그 용도다.

저널과 달리 해시는 버전 dispatch다. `CHAIN_HASH_VERSION`을 올리면 과거 행은 자기
버전으로 계속 검증된다. byte-stable 저널에는 없던 탈출구다.
</Decision>

원장을 새 바이너리로 열면 없는 컬럼을 읽고 죽었다. ledger.db는 사용자가 커밋해 둔
원장이고 플러그인 바이너리는 따로 올라간다. **플러그인을 한 번만 업데이트해도 기존
장부가 전부 안 열렸을 것이다.** 새 DB만으로 테스트했다면 놓쳤다.
</Decision>

<Decision
  id="ADR-005"
  title="스키마는 drizzle-kit. 쿼리는 우리 것"
  status="accepted"
  date="2026-07-17"
  commit="75a75e8"
  rejected={[
    { option: 'drizzle-orm 전면 도입 (쿼리 포함)', because: 'Drizzle의 SQLite 세션은 setReadBigInts(true)를 절대 호출하지 않고 integer()가 내부적으로 `number int53`으로 타이핑돼 있다. 2^53을 넘으면 ERR_OUT_OF_RANGE로 죽고 주입할 훅이 없다(prepareQuery 안의 지역 변수). 이 원장의 금액은 i64이고 적합성 스위트가 2^53+1 왕복을 못박는다.' },
    { option: "blob({mode:'bigint'})", because: 'bigint를 십진수 ASCII로 BLOB에 저장한다. 정확하지만 SUM()이 안 되고 ORDER BY가 바이트 정렬이라 getBalances가 통째로 날아간다.' },
    { option: '손으로 쓴 SQL 유지', because: '마이그레이션 관리가 원시적이었고, 무엇보다 "적용된 마이그레이션 수정 금지"가 강제되지 않는 주석이었다.' },
  ]}
>
DDL은 Drizzle이 만들고 읽기는 `Db` 래퍼가 한다. `integer()`가 내는 DDL은 평범한
`INTEGER`라 분리 비용은 0이다.

**도입을 밀던 전제는 틀렸다.** Drizzle은 SQLite와 Postgres에 스키마 하나를 주지
않는다. `sqliteTable`과 `pgTable`은 빌더가 다르고 컬럼 타입도 겹치지 않는다.
Supabase는 어차피 자기 스키마를 쓴다. SQLite `RAISE(ABORT)`와 Postgres plpgsql
`RAISE EXCEPTION`은 방언 차이지 툴체인 문제가 아니다. 포트 경계가 이미 허용한다.

**실제로 얻은 것은 해시 추적이다.** `__holiday_migrations`가 마이그레이션마다 sha256을
남긴다. 적용된 파일을 고치면 다음 열기에서 터진다. 아이디어는 Drizzle
`__drizzle_migrations`에서 왔지만 테이블은 우리 것이다. Drizzle `migrate()`는 런타임에
디스크 `.sql`을 읽고, 이 CLI는 옆에 아무것도 없는 단일파일이다.

**대가:** STRICT 테이블을 잃었다(drizzle-kit 미지원). INTEGER에 TEXT를 손편집으로
넣는 길을 막던 ring 3 하나가 빠졌다. RC(`1.0.0-rc.4`)에 핀됐다. stable에는
node:sqlite 드라이버가 없다.
</Decision>

<Decision
  id="ADR-006"
  title="스토어는 하나. 방언만 갈라진다"
  status="accepted"
  date="2026-07-17"
  commit="2c90bef"
  rejected={[
    { option: 'store-sqlite를 기계 변환해서 store-postgres 만들기 (착수 직전까지 갔다)', because: '쓰는 날에는 두 파일이 일치하지만, 한쪽에만 들어간 첫 손수정에서 갈라진다. 포트 경계를 둔 이유가 정확히 그걸 막으려는 건데 그 경계를 배신하는 방법이었다.' },
    { option: 'Drizzle로 스키마 하나 두고 두 엔진에 공유', because: 'ADR-005. sqliteTable과 pgTable은 별개 빌더고 트리거 본문은 진짜로 다른 SQL이다. 스키마는 공유가 불가능하다 — 하지만 그게 스토어까지 갈라야 한다는 뜻은 아니었다.' },
    { option: 'node:sqlite에 맞춰 스토어를 동기로 유지', because: 'Postgres를 동기로 만들 방법이 없다. 동기 엔진에 비동기 외투를 입히는 건 값이 싸고(모든 호출이 즉시 resolve된다) 반대 방향은 불가능하다.' },
  ]}
>
전제를 믿지 않고 **줄 수를 셌다.** 스토어 1,486줄 중 SQLite 고유인 것:

| | |
|---|---|
| `GLOB` | 3 |
| `PRAGMA` | 3 |
| `INSERT OR IGNORE` | 1 |
| `BEGIN IMMEDIATE` | 1 |
| **합계** | **8** |

여덟 줄이다. 구조적 차이는 `node:sqlite`가 동기라는 점 하나였다. 그건 **드라이버
인터페이스**를 두라는 근거지, 스토어를 하나 더 쓰라는 근거가 아니다.

그래서 `@holiday/store-sql`이 `LedgerStore`를 하나만 구현한다. 엔진 패키지는
드라이버·스키마·마이그레이션만 갖는다. 8곳은 `Dialect`로 접힌다 —
`subtreeWildcard()`, `prepare()`, 엔진의 선택적 `init`/`checkpoint` 훅.

**공유 SQL에 `GLOB`이 남은 건 SQLite-ism 유출이 아니다.** 매칭은 대소문자를 가려야
하는데 SQLite `LIKE`는 가리지 않는다. `Assets:bank:x`가 `Assets:Bank` 접두사에
걸린다. SQLite에서 그 의미의 철자가 `GLOB`이고, Postgres `LIKE`는 이미 대소문자를
가린다. 같은 의미, 방언만 둘. 드라이버가 옮긴다.

**리팩터를 판단 가능하게 한 것은 적합성 스위트다.** 1,486줄을 옮기고 전부 비동기로
바꿨는데 18개가 그대로 통과했다. 스위트가 없었다면 "아마 같을 것"이 최선이었다.

부산물로 타입체커가 잡아 준 것: `await this.db.all<T>(...).map(...)`은
`await (….map(…))`으로 파싱되어 `.map`이 Promise에 붙는다. 호출부마다 괄호가
필요했고, 20곳이 한 번에 드러났다.
</Decision>
