## 의존 방향

```
                  @holiday/core
              (도메인 + 포트 + 유스케이스)
                        ▲
        ┌───────────────┴───────────────┐
        │                               │
  @holiday/store-sql              store-testkit
  (LedgerStore 구현 — 딱 하나)      (적합성 스위트)
        ▲                               ▲
   ┌────┴─────┐                         │
   │          │                         │
store-sqlite  store-postgres            │
(드라이버 +   (드라이버 +               │
 스키마 +      스키마 +                 │
 PRAGMA)       plpgsql)                 │
   ▲          ▲                         │
   └────┬─────┴─────────────────────────┘
        │
   @holiday/cli
   (composition root — 어떤 엔진을 쓰는지 아는 유일한 패키지)
```

`core`는 어댑터에서 **아무것도** import하지 않는다. `sqliteLedgerStore()`를 아는 곳은
`cli`뿐이다.

**`LedgerStore` 구현은 엔진마다 두지 않는다. `store-sql`에 하나다.** 엔진 패키지가
맡는 건 드라이버·스키마·마이그레이션뿐이다. 취향이 아니라 측정 결과다 — 스토어
1,486줄 가운데 방언에 묶인 줄은 8줄이었다. [ADR-006](/docs/dev/adr) 참조.

**`@holiday/ports`를 따로 두지 않는다.** 헥사고날에서 포트 인터페이스는 도메인 소유다.
바깥과 말할 때 쓰는 도메인 어휘이기 때문이다. 패키지를 쪼개면 가짜 경계만 생기고
워크스페이스 엣지·버전 동기화 비용이 붙는다. `package.json` `exports` 서브패스로도
같은 강제력을 걸 수 있다.

## 포트는 불균형을 표현할 수 없다

핵심은 이것이다. `insertPosting()` 같은 API는 없다. 원자는 **검증된 거래 전체**다.

포트가 지켜야 할 불변식은 **원자성과 내구성뿐**이다. 균형을 어댑터마다 다시 검사하면
어댑터 개수만큼 서로 다른 "균형" 정의가 생긴다.

## 어댑터는 티어를 참칭할 수 없다

`init()`은 `tier === 'engine'`인데 세 보장이 전부 true가 아니면 **throw한다.**

**"노션은 원장이 될 수 없다"를 README 문장이 아니라 구조로 박아 둔다.** 노션에서
3-leg 거래는 API 3콜이다. 2번과 3번 사이에 실패하면 장부가 틀린 채로 남는다.
논거는 [ADR](/docs/dev/adr)에 있다.

`streamPostings()`(필수 프리미티브)와 `getBalances()`(capability 게이트 빠른 경로)를
짝지으면, 약한 백엔드는 **틀리지 않고** **느려진다.**

## 불변식 4중 링

| 링 | 위치 | 역할 |
|---|---|---|
| **1. 도메인 (권위)** | `Txn.create()` | 순수·동기·I/O 없음. 불균형 `ValidatedTxn`은 **반환 자체가 불가능**. mock 없이 property test 가능 |
| **2. 포트 (의무)** | `appendTxn()` | 의무는 원자성 + 내구성뿐. 비즈니스 규칙이 아님 |
| **3. 저장소 (심층방어)** | 트리거 (SQLite `RAISE(ABORT)` / Postgres plpgsql) | 링1 버그·캐스트·`sqlite3` 수동 편집을 잡음. engine 티어만. 불변식 11개, 두 엔진 공통 |
| **4. 감사** | `holiday verify` | 전체 스캔 + 해시 체인 |

포트는 일부러 **가장 약한** 링이다. 명세할 것은 기계적 보장(원자성)이지
비즈니스 규칙(균형)이 아니다.

## 적합성 스위트가 경계를 진짜로 만든다

TypeScript 인터페이스는 문서에 가깝다. **계약은 `runLedgerStoreConformance()`다.**
케이스는 18개고 **엔진마다** 돈다. 그중 셋이 위조하기 쉬운 지점을 못 박는다:

- unit of work 중간 예외 → **0행** 기록
- 캐스트로 밀어넣은 불균형 거래 → 저장소가 거부
- `2^53+1` 왕복이 `number`로 뭉개지지 않음

**첫 어댑터보다 먼저 작성했다.** 굽힐 압력이 생기기 *전에* 적합성 스위트가 있어야
경계가 정직하게 남는다.

값은 이미 나왔다. `store-sql` 추출은 1,486줄을 패키지 밖으로 옮기고 동기를 전부
비동기로 바꾼 리팩터였는데, 18개가 **그대로** 통과했다. 스위트가 없었다면 "아마
같을 것"이라고밖에 말할 수 없었다. Postgres 어댑터도 같은 18개로 증명한다. pglite는
목이 아니다. WASM으로 컴파일된 진짜 Postgres라 트리거와 int8도 그대로 살아 있다.

## 배포 제약이 어댑터를 결정했다

플러그인은 **설치 때 빌드하지 않는다.** 캐시에 복사될 뿐이라 `pnpm install`도
postinstall도 없다. 커밋된 파일이 곧 실행 파일이다.

그래서 `better-sqlite3` 대신 **`node:sqlite`**를 쓴다. 네이티브 애드온은 번들되지 않고
플랫폼별 prebuild나 vendored `node_modules`를 플러그인에 넣는 일도 불가능하다.
builtin이면 순수 JS로 번들된다.

`plugin/bin/holiday.mjs`는 커밋된 빌드 산출물이다. 추하고 금방 썩는다. CI가 재번들한 뒤
커밋본과 diff한다.

## 감사

`audit_log`는 해시 체인이다. 각 행은 `prev_hash`로 직전 행에 커밋한다. `txn_append`
행은 **거래 id가 아니라 내용 sha256에 커밋한다.** 손으로 전기를 고치면 잔액·통화·
identity weight는 모두 맞아 다른 검사를 통과한다. **알아채는 건 체인뿐이다.**

**정직한 한계:** 우발적 변조는 잡는다. 쓰기 권한자가 체인 전체를 다시 계산하는 공격은
막지 못한다. head를 파일 **밖**에 앵커해야 닫힌다. 그 용도가 `holiday verify --head`다.
