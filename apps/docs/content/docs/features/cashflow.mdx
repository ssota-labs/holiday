원장이 아는 건 **과거뿐**이다. "다음달 카드값 나가면 통장이 버티나"에는 답하지 못한다.
스케줄은 반대로 본다 — **언제 돈이 나갈지**.

## 유지하는 게 아니라 유도한다

현금흐름표의 **과거 절반은 쿼리다** — 현금 계정을 건드린 posting이라 원장과 어긋날
여지가 없다. 손으로 맞춘 두 테이블은 언젠가 반드시 어긋난다.

계산이 필요한 쪽은 **미래 절반**뿐이고, 그건 스케줄에서 뽑는다.

## 네 개는 같다

카드·할부·대출은 회계상 **전부 부채**다. 갈리는 지점은 계정 종류가 아니라 **스케줄
모양**이다. 정기지출은 계정조차 아니다.

| | 모양 | 계정 |
|---|---|---|
| **카드 청구주기** | 회전 — 사용기간이 닫히고 결제일에 전액 | `Liabilities:Card:*` |
| **할부** | 고정 N개월 분할 | `Liabilities:Card:*:Installment` ← **별도** |
| **대출** | 분할상환 (원금+이자) | `Liabilities:Loans:*` |
| **정기지출** | 반복 발생 | 계정 아님 |

**스케줄에서 원장으로 가는 화살표가 없다.** 예측은 절대 전기되지 않는다. 금리나
구독이 바뀔 때마다 과거가 오염되기 때문이다.

## 실제 출력

```
cash on hand (2026-07-17):  350000000 KRW

2026-07-25   -      800000   →     349200000
             월세                                   800000
2026-08-01   -      400000   →     348800000
             신한  2026-06-15..2026-07-14           400000
2026-08-25   -     2267052   →     346532948
             월세                                   800000
             KB 주담대 (1/360)                      1467052
2026-09-01   -      100000   →     346432948
             냉장고 (1/12)                           100000
```

8/25에 월세와 주담대가 **같은 날 합류**한다. 통장이 실제로 하는 일과 같다.

답이 `⚠ SHORT`다. 잔액은 숫자 하나지만, 현금흐름은 **어느 날 터지는지**를 보여 준다.

## 현금이 뭔지는 플래그가 정한다

기준은 `account.cash`다. 예전에는 `Assets:Bank*` 접두어 규칙이었고, 관례를 사실인 척했다.
다른 이름 아래 현금이 투영에서 **조용히** 빠졌는데도 숫자는 멀쩡한 척 틀렸다. [ADR-004](/docs/dev/adr#adr-004).

표시되지 않은 자산 계정이 있으면 `cashflow`가 소리 내어 알린다. 예측에서 항목이 조용히
빠지면 "모르겠다"가 "괜찮다"로 읽힌다.

## 스케줄은 예측이다 — 근사가 허용되는 유일한 곳

원장의 "허용오차 없음"과 모순처럼 보여도 아니다. **사실/예측을 나눈 반대편**일 뿐이다.

- **사실**(원장): 정확. 허용오차 0. `SUM(weight) = 0`.
- **예측**(스케줄): 근사 가능. `loan check`가 정직하게 유지한다.

다만 **행의 합만은 정확**해야 한다. 합이 안 맞는 스케줄은 명세서와 영원히 대사되지 않고,
예측을 정직하게 유지할 수단도 사라진다.
