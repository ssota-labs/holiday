> **지켜야 하는 규칙:** [POLICY-001](/docs/domain/policy#policy-001)
> 동작은 여기 두고, 항상 참이어야 할 규칙은 [도메인 정책](/docs/domain/policy)에 있다.

CLI는 **OCR을 하지 않는다.** 에이전트 vision 모델이 이미지를 읽어 파싱 JSON을 넘기면,
여기서는 새 항목인지 가려 내고, 리뷰 게이트에서는 맞는지 확인한다.

분리가 핵심이다. **모델은 이 시스템에서 가장 신뢰도가 낮은 부품**이다. 나머지 코드는
그 실수를 되돌릴 수 있게 두기 위해 있다.

## 드래프트지 별도 staging이 아니다

인제스트가 쓰는 건 `status='draft'`인 **진짜 거래**다. 균형 규칙이 인제스트 시점에
바로 돌아가니 **불균형 드래프트는 애초에 만들 수 없다.**

따라오는 결과는 둘이다. 망가진 항목을 큐에 쌓아 남에게 떠넘길 길이 없고, 뒤이은 승인은
**실패할 수 없는 status flip**이 된다.

**`--idem-key`는 선택이 아니다.** CLI를 shell out하는 에이전트는 타임아웃이 나면 **반드시**
재시도한다. 같은 키·같은 요청은 replay하고, 같은 키·다른 요청은 하드 에러다.
이중 기표는 원장 신뢰를 깨뜨리는 유일한 실패 모드다.

## 중복 판단은 권위가 층으로 나뉜다

처음 계획은 자연키(`계정‖날짜‖금액‖상호`)에 `UNIQUE`를 걸어 막으면서, 동시에
"같은 날 같은 카페 ₩4,500 두 잔은 평범한 화요일"이라고도 했다. **둘은 함께 참일 수 없다.**
그 두 잔의 자연키는 정확히 같고, 막으면 진짜 거래까지 거절한다.

| 근거 | 판단 | 왜 |
|---|---|---|
| **이미지 sha256** | 차단 | 같은 바이트는 항상 같은 캡쳐다. 틀릴 수가 없는 유일한 검사 |
| **externalRef** | 차단 | 카드사 거래ID보다 카드사 기록에 권위 있는 건 없다 |
| **자연키** | **경고** | 추측이다. 커피를 두 번 사는 바로 그때 틀린다 |
| **±3일 퍼지** | **경고** | 카드 앱과 명세서가 날짜를 다르게 말한다 |

상호명 정규화는 **의도적으로 보수적으로** 둔다. 카드사가 붙인 법인 형태(`(주)`, `주식회사`)와
네트워크 구분자는 떼되 **지점은 남긴다.** `GS25 역삼점`과 `GS25 강남점`은 다른 곳이고,
합치면 실제 구매 둘이 하나로 보인다.

## 거부해도 기록은 남는다

거부된 항목은 **지우지 않는다.** 그건 중복 판단의 기억이다. 지우면 같은 캡쳐가
다시 올라온다.

```bash
holiday review list
holiday review accept <id>              # 균형은 이미 잡혀 있으니 실패할 수 없다
holiday review reject <id> --reason "…"  # 기록은 남는다
```

## 게이트는 추측 허가증이 아니다

리뷰가 잡는 대상은 **잘못된 카테고리**이지 **오독한 금액**이 아니다. 사람이
`₩1,240,00`을 승인하면 틀린 채로 남는다.

스킬이 "금액이 불분명하면 멈추고 물어라"에 지면을 할애하는 까닭이다. 게이트가 생겼다고
모델이 느슨해지면 게이트 없는 편이 낫다.
