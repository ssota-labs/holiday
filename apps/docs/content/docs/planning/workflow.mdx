## 1. 변경을 분류한다

| 변경 | 필요한 문서 |
|---|---|
| 새 기능·사용자 동작 변경 | PRD + 스펙 + 구현계획 |
| 기존 계약에 맞추는 버그 수정 | 기존 PRD·스펙 + 구현계획 |
| 리팩터링·빌드·의존성 | 구현계획 |
| docs-only | 선행 계획 면제 |

기존 문서가 변경 이유와 계약을 충분히 설명하면 새 문서를 만들지 않고 ID로 참조한다.

## 2. 누구에게 쓰는 문서인가

기획 문서(PRD·User Story·기능 수락 기준)의 **첫 독자는 기획자·리뷰어·비개발 팀원**이다.
구현 에이전트와 개발자는 두 번째 독자다.

리뷰에서 물을 질문이 이것이다.

- 무엇을 남기거나 / 무엇을 보여 주나?
- 사람이 아는 말로 필드·칸·상태가 설명되어 있나?
- 수락 기준을 제품 화면·신고서·통장 기준으로 체크할 수 있나?
- 개발 용어(`append`, `header`, `revision`, 테이블명)만으로 본문을 때우지 않았나?

통과하지 못하면 구현으로 넘어가지 않는다. 기계 검증은 **순서·참조 그래프**만 보고,
이 판단은 사람이 한다.

### 작성 기준

1. **한 줄로** 무엇을 하는지 먼저 쓴다.
2. **저장·조회하는 데이터**를 표로 푼다. 사람 이름 → 예시 값 → 왜 필요한가.
3. 코드/스키마 용어를 쓸 때는 바로 옆에 한국어 설명을 붙인다.
   (`revision` = 신고 차수, 최초·수정을 덮어쓰지 않고 번호로 남긴다)
4. 수락 기준은 **관찰 가능한 결과**로 쓴다.
   - 나쁨: `헤더+라인 append (POLICY-018)`
   - 좋음: `신고 표지와 세액표 숫자(소득금액·과세표준·납부할세액)를 한 번에 저장한다`
5. 구현 제약(bigint, 마이그레이션, 패키지 경로)은 **별도 절**에 두고, 기획 본문과 섞지 않는다.
6. CLI 플래그·JSON 키·POLICY ID는 스펙·정책 문서에 두고, PRD/US에서는 링크로 연결한다.

템플릿: [`apps/docs/templates`](https://github.com/ssota-labs/holiday-cfo/tree/main/apps/docs/templates)
좋은 예: [prd-tax-return-sor](/docs/planning/prds/prd-tax-return-sor)

## 3. 기획 PR을 먼저 병합한다

템플릿을 복사해 PRD·스펙·구현계획을 채운다. 문서 참조가 완성되면 PRD는 `active`,
스펙은 `accepted`, 구현계획은 `ready`로 바꾼다.

기획 PR에는 제품 구현 코드를 섞지 않는다. 기획 PR을 올린 뒤 **사람 리뷰·승인(또는
기본 브랜치 병합) 전에는 구현을 시작하지 않는다.**

## 4. 구현 PR에서 계획을 참조한다

PR 본문의 `Plan:`에 저장소 기준 구현계획 경로를 적는다.

`Plan: apps/docs/content/docs/planning/plans/plan-<initiative>.mdx`

검증기는 해당 파일이 PR의 base SHA에 이미 있는지 확인한다. 같은 PR에서 계획을 새로
추가하면 통과하지 않는다.

## 5. 계획대로 검증하고 상태를 닫는다

구현 중 범위나 계약이 달라지면 기획 PR을 먼저 갱신한다. 구현과 검증이 끝나면 계획을
`done`으로 바꾼다. PRD는 포함된 계획과 스토리가 모두 끝났을 때 `done`으로 바꾼다.

## 로컬 검증

```bash
pnpm --filter @holiday-cfo/docs check:planning   # 참조 그래프·ID
pnpm check:docs-first                            # 구현 PR Plan: 선행 (있을 때)
pnpm --filter @holiday-cfo/docs test
pnpm --filter @holiday-cfo/docs build
```

기계 검증은 구조와 순서만 보장한다. 목표·비목표·수락 기준이 **비개발자에게 읽히는지**는
기획 PR 리뷰에서 판단한다.
